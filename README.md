# AutoSlot

AutoSlot — MVP-приложение на **Elixir / Phoenix LiveView** для онлайн-записи клиентов в небольшой автосервис.

Проект моделирует реалистичный бизнес-кейс: клиент выбирает услугу, дату и свободное время, вводит контактные данные и номер автомобиля, после чего система создает запись. Сотрудник автосервиса управляет заявками через административную панель, а клиент может самостоятельно найти свои записи по номеру телефона и отменить активную запись.

## Зачем нужен проект

Небольшие автосервисы часто принимают записи вручную: через телефонные звонки, мессенджеры или таблицы. Такой подход работает на старте, но быстро создает проблемы:

- сотрудники тратят время на однотипные вопросы о свободном времени;
- клиент не может самостоятельно увидеть доступные слоты;
- возможны ошибки и двойные записи;
- длительность услуги приходится учитывать вручную;
- историю записей сложно анализировать;
- клиент зависит от администратора, даже если хочет просто проверить статус записи.

AutoSlot решает эту задачу через простой онлайн-сценарий записи и базовую административную панель.

## Основной пользовательский сценарий

1. Клиент открывает страницу онлайн-записи.
2. Выбирает услугу из каталога.
3. Выбирает дату.
4. Система показывает только доступные временные слоты.
5. Клиент вводит имя, телефон и номер автомобиля.
6. Система создает запись со статусом `pending`.
7. Клиент может найти свои записи по телефону на странице "Мои записи".
8. Администратор подтверждает или отменяет запись в панели управления.

## Реализованная функциональность

### Каталог услуг

В проекте реализован каталог услуг автосервиса. Для каждой услуги хранится название, описание, длительность в минутах и ориентировочная цена.

Каталог используется не только как справочник, но и как часть бизнес-логики: длительность выбранной услуги влияет на расчет свободных временных слотов.

### Клиентская запись

Клиент может создать запись на странице `/book`.

Форма записи включает:

- выбор услуги;
- выбор даты;
- выбор свободного временного слота;
- имя клиента;
- телефон;
- номер автомобиля.

После создания запись получает статус `pending`.

### Карточка выбранной услуги

На странице записи отображается карточка выбранной услуги.

Клиент видит:

- название услуги;
- длительность;
- цену;
- описание.

Это помогает клиенту проверить выбранную услугу перед отправкой формы.

### Расчет свободных слотов

Система рассчитывает доступные временные интервалы с учетом:

- выбранной даты;
- длительности услуги;
- рабочего дня;
- уже существующих активных записей.

Активные записи со статусами `pending` и `confirmed` блокируют слот. Отмененные записи со статусом `cancelled` не блокируют время.

### Защита от пересечений

В бизнес-логике реализована защита от двойной записи на одно и то же время.

Система не позволяет создать или обновить активную запись так, чтобы она пересекалась по времени с другой активной записью.

### Запрет дат в прошлом

На странице онлайн-записи нельзя выбрать дату раньше текущей даты.

Дополнительно LiveView нормализует некорректную прошлую дату обратно к сегодняшней, если ее попробовать передать вручную.

### Поиск клиентских записей

На странице `/my-bookings` клиент может найти свои записи по номеру телефона.

Страница показывает:

- услугу;
- дату и время записи;
- имя клиента;
- телефон;
- номер автомобиля;
- текущий статус записи.

### Отмена записи клиентом

Если запись активна, клиент может отменить ее через страницу `/my-bookings`.

При отмене запись не удаляется из базы, а получает статус `cancelled`. Это сохраняет историю и позволяет администратору видеть, что заявка была отменена.

### Административная панель

На странице `/admin/bookings` реализована административная панель для сотрудников автосервиса.

Возможности панели:

- просмотр записей по дате;
- фильтрация по статусу;
- поиск по имени клиента, телефону, номеру автомобиля и услуге;
- сводные карточки по количеству записей;
- подтверждение записи;
- отмена записи;
- модальное подтверждение перед отменой.

### Статусы записи

В проекте используются три основных статуса:

| Статус | Значение |
| --- | --- |
| `pending` | Запись создана и ожидает подтверждения |
| `confirmed` | Запись подтверждена администратором |
| `cancelled` | Запись отменена |

## Локальный запуск

### Требования

Для запуска проекта локально нужны:

- Elixir;
- Erlang/OTP;
- PostgreSQL;
- Git.

### Клонирование репозитория

```bash
git clone https://github.com/takeshi-katana/autoslot.git
cd autoslot
```

### Установка зависимостей

```bash
mix deps.get
```

### Создание базы данных

```bash
mix ecto.create
```

### Выполнение миграций

```bash
mix ecto.migrate
```

### Заполнение тестовыми данными

```bash
mix run priv/repo/seeds.exs
```

### Запуск сервера

```bash
mix phx.server
```

После запуска приложение будет доступно по адресу:

```text
http://localhost:4000
```

## GitHub Actions CI

В проекте используется GitHub Actions для автоматической проверки Pull Request.

CI проверяет:

- форматирование кода;
- компиляцию проекта;
- прохождение тестов.

## GitHub Workflow

Проект ведется через простой GitHub-процесс:

1. Создается issue под задачу.
2. Создается отдельная feature-ветка.
3. Ветка получает осмысленное название.
4. Реализуется изменение.
5. Выполняются локальные проверки.
6. Создается commit с понятным сообщением.
7. Открывается Pull Request.
8. GitHub Actions запускает CI.
9. После успешной проверки PR мержится в `main`.

Примеры веток:

```text
feature/service-catalog-polish
feature/customer-booking-lookup
feature/link-customer-bookings
feature/link-my-bookings-from-booking-page
feature/show-selected-service-summary
feature/prevent-past-booking-dates
chore/format-project-files
```

В текущую версию намеренно не включены:

- онлайн-оплата;
- SMS-уведомления;
- WhatsApp-уведомления;
- интеграция с CRM;
- полноценная авторизация сотрудников;
- разделение ролей пользователей;
- production-деплой;
- аналитика и отчеты.

Эти функции можно добавить на следующих этапах, но для MVP основной фокус сделан на логике записи, управлении заявками и проверяемой бизнес-логике.
