# Architecture

AutoSlot построен как Phoenix-приложение с использованием PostgreSQL для хранения данных.

Архитектура проекта следует стандартному подходу Phoenix: веб-слой отделен от бизнес-логики, а работа с базой данных выполняется через Ecto.

## Общая схема

```text
Браузер клиента
      |
      v
Phoenix Web Layer
      |
      v
Application Contexts
      |
      v
Ecto Repo
      |
      v
PostgreSQL
```

## Основные слои приложения

## 1. Web Layer

Web Layer отвечает за взаимодействие с пользователем.

В этот слой входят:

- страницы;
- формы;
- LiveView;
- контроллеры;
- компоненты;
- отображение ошибок валидации;
- обработка пользовательских действий.

Файлы веб-слоя находятся в директории:

```text
lib/autoslot_web/
```

Главное правило: веб-слой не должен содержать сложную бизнес-логику. Он должен вызывать функции контекстов из основного слоя приложения.

## 2. Context Layer

Context Layer содержит бизнес-логику приложения.

Файлы контекстов находятся в директории:

```text
lib/autoslot/
```

Планируемые контексты:

```text
Autoslot.Services
Autoslot.Bookings
Autoslot.Scheduling
```

### Autoslot.Services

Контекст `Services` отвечает за услуги автосервиса.

Задачи:

- создание услуги;
- получение списка услуг;
- получение конкретной услуги;
- изменение услуги;
- удаление услуги при необходимости.

### Autoslot.Bookings

Контекст `Bookings` отвечает за записи клиентов.

Задачи:

- создание записи;
- получение списка записей;
- фильтрация записей по дате;
- изменение статуса записи;
- отмена записи;
- проверка корректности данных записи.

### Autoslot.Scheduling

Контекст `Scheduling` отвечает за расчет доступного времени.

Задачи:

- генерация временных слотов;
- учет длительности услуги;
- проверка занятых интервалов;
- исключение недоступных слотов;
- предотвращение пересечений записей.

## 3. Persistence Layer

Persistence Layer отвечает за хранение данных.

В проекте используется:

```text
Ecto + PostgreSQL
```

Файлы, связанные с базой данных, находятся в директории:

```text
priv/repo/
```

Там располагаются:

- миграции;
- seed-данные;
- структура базы данных.

## Планируемая модель данных

## Service

`Service` описывает услугу автосервиса, которую может выбрать клиент.

Планируемые поля:

```text
id
name
description
duration_minutes
price
inserted_at
updated_at
```

Описание полей:

```text
name              название услуги
description       описание услуги
duration_minutes  длительность услуги в минутах
price             ориентировочная стоимость услуги
```

Примеры услуг:

```text
Компьютерная диагностика
Замена масла
Шиномонтаж
Ремонт тормозной системы
Диагностика подвески
```

## Booking

`Booking` описывает запись клиента на услугу.

Планируемые поля:

```text
id
customer_name
phone
vehicle_plate
starts_at
ends_at
status
service_id
inserted_at
updated_at
```

Описание полей:

```text
customer_name  имя клиента
phone          телефон клиента
vehicle_plate  номер автомобиля
starts_at      время начала записи
ends_at        время окончания записи
status         статус записи
service_id     ссылка на выбранную услугу
```

Планируемые статусы:

```text
pending
confirmed
cancelled
```

## Связь между таблицами

Одна услуга может иметь много записей.

```text
Service 1 ---- * Booking
```

То есть:

- `services` хранит список услуг;
- `bookings` хранит записи клиентов;
- каждая запись связана с одной услугой через `service_id`.

## Правила расписания

Начальные правила планирования:

```text
Рабочий день: 09:00–18:00
Шаг слота: 30 минут
Отмененные записи не блокируют время
Активные записи не должны пересекаться
Длительность услуги влияет на доступные слоты
```

## Пример расчета слотов

Если услуга длится 60 минут, а шаг слота равен 30 минутам, система может предложить такие варианты:

```text
09:00–10:00
09:30–10:30
10:00–11:00
10:30–11:30
```

Если уже есть запись:

```text
10:00–11:00
```

то система не должна предлагать слоты, которые пересекаются с этим временем:

```text
09:30–10:30
10:00–11:00
10:30–11:30
```

## Проверка пересечений

Две записи считаются пересекающимися, если время одной записи попадает внутрь времени другой записи.

Базовое правило:

```text
new_start < existing_end AND new_end > existing_start
```

Пример:

```text
Существующая запись: 10:00–11:00
Новая запись:        10:30–11:30
```

Такая запись должна быть запрещена, потому что интервалы пересекаются.

## Тестирование

Самая важная часть проекта — бизнес-логика записи и расписания.

Планируемые тесты:

- создание услуги;
- валидация услуги;
- создание записи;
- валидация записи;
- расчет свободных слотов;
- исключение занятых слотов;
- запрет двойной записи;
- изменение статуса записи;
- отмена записи.

## GitHub Actions

Для проекта планируется CI-процесс через GitHub Actions.

Минимальная проверка должна включать:

```text
mix deps.get
mix compile --warnings-as-errors
mix test
```

В дальнейшем можно добавить:

```text
mix format --check-formatted
mix credo
```
